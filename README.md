## Why This Exists

Search is changing.

Traditional SEO focused heavily on keywords, links, and pages. Modern search systems also need to understand entities, relationships, context, services, authority signals, structured information, and trust.

AI systems do not only return links. They retrieve, summarize, compare, recommend, and cite information.

That means businesses need more than content. They need clear digital infrastructure that explains who they are, what they do, who they serve, what they know, where they operate, what services they provide, and why they are credible.

Rakos Media Group uses this repository to organize that public knowledge in a clear, structured, and machine-readable way.

## Repository Structure

The repository is organized into folders that separate entities, frameworks, services, concepts, industries, playbooks, templates, scripts, and schemas.

The `entities` folder defines organizations, people, brands, historical names, and named assets connected to Rakos Media Group.

The `frameworks` folder explains Rakos Media Group methodologies and public-facing strategic frameworks.

The `services` folder explains what Rakos Media Group offers and how each service supports visibility, trust, conversion, growth, or machine understanding.

The `concepts` folder defines important topics such as entity SEO, knowledge graphs, structured data, schema markup, AI visibility, GEO, AEO, local SEO, technical SEO, and machine-readable business infrastructure.

The `industries` folder explains how visibility, search, websites, ads, AI systems, and machine-readable infrastructure apply to specific business categories.

The `playbooks` folder provides high-level educational guidance. Playbooks explain what matters and why without exposing private internal workflows.

The `templates` folder defines the structure for future knowledge documents.

The `scripts` folder may support validation, formatting, schema generation, internal link checks, or publishing workflows.

The `schemas` folder supports structured data and machine-readable publishing.

## Markdown and YAML Standard

Knowledge files use Markdown with YAML frontmatter.

The current YAML standard uses inline arrays to reduce formatting errors.

Public GitHub files should use `status: Published`.

The repository should not use `website_status`.

YAML frontmatter should not use Markdown star bullets.

After the YAML block, each knowledge file should begin with one H1 title.

The public body content should use clear headings and paragraphs. Bullet lists should be avoided unless they are truly necessary.

## Public Knowledge Rules

This repository is designed to publish the what and the why, not the full private how.

Public knowledge may include definitions, concept explanations, framework overviews, FAQs, thought leadership, industry education, high-level strategy, public service descriptions, public entity descriptions, and general educational guidance.

This repository should not publish private prompts, internal SOPs, client strategies, client data, audit rubrics, scoring systems, automation logic, API keys, passwords, proprietary workflows, internal decision trees, or private implementation details.

The guiding principle is simple.

Publish the what and the why.

Protect the exact how.
