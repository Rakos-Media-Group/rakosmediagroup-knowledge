# Contributing Guidelines

This repository contains public knowledge files for Rakos Media Group.

The goal is to create clear, structured, machine-readable content that helps people, search engines, AI search systems, answer engines, and future agents understand Rakos Media Group, its services, frameworks, concepts, and areas of expertise.

Rakos Media Group builds machine-readable businesses.

## Purpose of This Repository

This repository supports public knowledge development around:

* SEO
* AI search optimization
* Google Ads
* Paid media
* Website development
* Conversion strategy
* Entity SEO
* Knowledge graph engineering
* Structured data
* Schema markup
* Search Everywhere visibility
* Machine-readable business infrastructure

Every file should help clarify one or more of these areas.

## Writing Standards

All content should be written in clear English.

Use:

* Clear headings
* Short paragraphs
* Helpful explanations
* Internal relationships
* FAQs when useful
* Public-facing language
* Plain, confident explanations

Avoid:

* Jargon without explanation
* Overly technical language where simple language works
* Sales-heavy copy
* Vague claims
* Hidden methods
* Private workflows
* Unverified claims
* Em dashes

## File Format

Most knowledge files should use Markdown with YAML frontmatter.

Use this standard structure unless there is a good reason to adjust it:

```yaml
---
title:
slug:
type:
status:
primary_entity:
schema_type:
word_count_target:
related_services:
related_concepts:
related_frameworks:
related_industries:
source_urls:
last_updated:
---
```

## Allowed File Types

Use one of the following content types:

* Entity
* Framework
* Service
* Concept
* Industry
* Playbook

## Folder Guidelines

Use the correct folder for each file.

```text
entities/
frameworks/
services/
concepts/
industries/
playbooks/
templates/
scripts/
schemas/
```

Examples:

* Opportunity Atlas™

* Conversion Compass™

### services

Use for public service categories.

Examples:

* SEO
* AI search optimization
* Google Ads management
* Website development
* Knowledge graph engineering

### concepts

Use for educational topic definitions.

Examples:

* Entity SEO
* Knowledge graphs
* Structured data
* Schema markup
* GEO
* AEO

### industries

Use for industry-specific knowledge pages.

Examples:

* Manufacturing marketing
* Healthcare marketing
* Hospitality marketing
* Ecommerce marketing
* Wellness marketing

### playbooks

Use for high-level public guidance.

Playbooks may explain what matters, why it matters, and what a business should think about.

Do not include private step-by-step internal workflows.

## Content Requirements

Each knowledge file should include:

1. YAML frontmatter
2. Clear title
3. Definition or overview
4. Why the topic matters
5. How Rakos Media Group thinks about the topic
6. Related services
7. Related concepts
8. Related frameworks when relevant
9. FAQ section when useful
10. Suggested schema type
11. Internal link suggestions

## Internal Linking Guidelines

Every file should suggest internal relationships.

Look for links between:

* Services and concepts
* Concepts and frameworks
* Frameworks and services
* Industries and services
* Entities and frameworks
* Entities and services

Example relationships:

* Rakos Media Group owns Opportunity Atlas™
* Opportunity Atlas™ informs Search Everywhere Framework™
* Search Everywhere Framework™ supports Search Everywhere Graph™
* Search Everywhere Graph™ supports knowledge graph engineering
* Knowledge graph engineering supports AI search optimization
* AI search optimization supports visibility in Google and AI systems

## Public Knowledge Rules

This repository should publish the what and the why.

It should not publish the full private how.

### Safe to Publish

* Definitions
* Concepts
* Framework overviews
* FAQs
* Thought leadership
* Industry education
* High-level strategy
* Public service descriptions
* Public entity descriptions

### Do Not Publish

* Private prompts
* Internal SOPs
* Client strategies
* Client data
* Audit rubrics
* Scoring systems
* Automation logic
* API keys
* Passwords
* Private workflows
* Internal decision trees
* Internal research process
* Proprietary implementation details

## Review Checklist

Before committing a new file, review these questions:

1. Does the file have YAML frontmatter?
2. Is the title clear?
3. Is the content written in plain English?
4. Does it avoid em dashes?
5. Does it explain the topic clearly?
6. Does it include related services, concepts, or frameworks?
7. Does it avoid private prompts?
8. Does it avoid client data?
9. Does it avoid internal scoring systems?
10. Does it avoid proprietary workflows?
11. Does it suggest a schema type?
12. Does it support Rakos Media Group entity clarity?

## Tone

The tone should feel:

* Clear
* Strategic
* Useful
* Professional
* Confident
* Human
* Search-aware
* AI-aware

Avoid sounding like generic SEO content.

Rakos Media Group’s point of view should be clear:

The future of search belongs to businesses that are clear enough for humans to trust and structured enough for machines to understand.

## Brand Naming

Use:

* Rakos Media Group as the primary entity name
* Rakos Media as the public shorthand
* rakosmediagroup.com as the domain anchor

Do not use Rakos alone unless it appears naturally in a sentence and the meaning is clear.

## Final Principle

Make Rakos Media Group easy for machines to understand without giving away the private system that makes the work valuable.
